# Milestone v0.1 — first usable prototype

End state: an operator can write a `nebula.hcl`, run `nebula-pki`, and get one or more CAs plus host certificates on disk, fanned out to multiple directories, with a trust bundle for `pki.ca`, an idempotent manifest, declarative CA rotation, time-based renewal, and air-gapped (`in_pub`) signing. Installable via Homebrew and Nix. Encryption explicitly **out of scope** for v0.1.

Versioning: everything between today and the v0.1 cut ships under `0.0.x` tags. Each `0.0.x` is a real release: tagged, built, installable. Only when every v0.1 acceptance criterion below is met do we cut `0.1.0`.

## Acceptance criteria for v0.1.0

The CLI implements, with tests, the following from [`spec/`](../readme.md):

- `ca {}` generate mode — writes `ca.crt` / `ca.key` at default or configured paths.
- `ca { cert_file, key_file }` reference mode — uses existing CA, never rewrites it.
- Multiple labelled `ca "<label>" {}` blocks, `host.ca`, and `ca.default` — signing-CA selection per host ([ADR-015](../adr/015-multiple-cas-per-config.md)).
- `host "<label>" {}` signing — generates keypair, signs cert with the upstream `slackhq/nebula/cert` library.
- `host.in_pub` air-gapped signing — signs a device-supplied public key, writes cert only, no key, no encryption ([ADR-018](../adr/018-in-pub-air-gapped-signing.md)).
- `host.output_dir` — per-host destination directory; cert/key filenames derived from cert `name`. `out_crt` / `out_key` override the path component within the directory and compose with `output_dir`.
- Trust bundle emission at `out/ca/bundle.crt` (renameable) — concatenated active CA certs, always written ([ADR-016](../adr/016-ca-rotation-and-trust-bundles.md)).
- Declarative CA rotation — staging a second CA, moving the `default = true` marker, archiving the old CA; re-sign driven by `ca_fingerprint` change ([ADR-016](../adr/016-ca-rotation-and-trust-bundles.md)).
- `renew_before` (host and CA level) — re-sign within the window of expiry ([ADR-017](../adr/017-host-renewal-threshold.md)).
- Post-run deadline reporting — after reconcile/`--dry-run`, print the earliest actionable renewal/expiry deadline and a "run again before <date>" hint ([ADR-017](../adr/017-host-renewal-threshold.md)).
- Manifest at `out/nebula-pki.json` with `schema_version = 1`, conforming to [ADR-002](../adr/002-state-and-artifact-layout.md), including `cas`, `trust_bundle`, per-host `ca` / `renew_before` / `in_pub`.
- `nebula-pki check` — parse + validate, no I/O against `out/`.
- `nebula-pki --dry-run` — preview without writes.
- Idempotency — second run against unchanged config writes nothing (given a fixed clock and no host inside its renewal window).
- Installable via `brew install anverse/nebula-pki/nebula-pki` and `nix run github:anverse/nebula-pki`.

Explicitly **not** in v0.1:

- `storage.encryption` block (any backend). Parser may accept the block but the CLI errors with a clear "encryption ships in v0.2" message rather than silently ignoring it.
- `nebula-pki show`.
- Anything in the [non-goals or deferred sections](../readme.md#non-goals) of the spec.

## Code structure concept

Layout follows standard Go layout, kept small. Internal packages only — no public API surface yet.

```
cmd/nebula-pki/        # thin main; wires cobra/flag + version + entrypoint
internal/
  config/              # HCL parsing + schema-level validation (pure, no I/O)
  pki/                 # CA generate/reference + host signing (wraps slackhq/nebula/cert)
  manifest/            # nebula-pki.json read/write, diffing for idempotency
  plan/                # turn config + manifest + filesystem state into a list of actions
  apply/               # execute a plan; --dry-run = build plan, print, exit
  fsutil/              # atomic write, dir create, path resolution
  cli/                 # cobra commands: root (reconcile), check, version
internal/buildinfo/    # version, commit, date — populated by ldflags
test/
  e2e/                 # testscript scenarios (*.txtar)
  testdata/            # shared HCL fixtures, golden manifests
```

Separation of concerns ([principles](../../agents.md)):

- `config` is pure: bytes in → struct + errors out. No filesystem.
- `pki` is pure on the input side: takes parsed config + bytes from existing CA files, returns cert/key bytes. No filesystem writes.
- `manifest` is pure on the data side: reads/writes JSON, but doesn't decide what should change.
- `plan` is the only place that compares "what's on disk + manifest" to "what the config asks for" and produces a list of `Action`s (`WriteFile`, `Noop`, `Skip`).
- `apply` is the only place that mutates the filesystem.
- `cli` only glues these together.

This shape makes idempotency a first-class testable property: `plan(config, current_fs) → []Action`; assert `len == 0` after one successful apply.

### Library choices

- HCL: `github.com/hashicorp/hcl/v2` with `gohcl` for struct decoding. No `EvalContext` — see [ADR-005](../adr/005-hcl-schema-decision.md).
- CLI: `github.com/spf13/cobra` (mature, supports `--dry-run`, subcommands cleanly).
- Cert primitives: `github.com/slackhq/nebula/cert` — same library `nebula-cert` itself uses.
- Logging: stdlib `log/slog`. A `--log-format` flag selects `text` (default, human-friendly on a TTY) or `json` (structured, for CI / log shippers). A `--log-level` flag selects `debug|info|warn|error`. The configured handler is built once in `cmd/nebula-pki` and threaded through as `*slog.Logger`; no package-level globals.
- JSON: stdlib `encoding/json`.
- Diffing in tests: `github.com/google/go-cmp/cmp`.
- E2E: `github.com/rogpeppe/testscript` (slim fork of `go-internal/testscript` with fewer transitive dependencies).

Everything else: stdlib.

## Testing strategy

Two layers, each justified by what it catches:

### Unit tests (per package, in-package)

- `config/`: table-driven HCL fixtures → expected parsed struct / expected validation error. Covers every validation rule listed in [`spec/hcl-schema.md`](../hcl-schema.md#validation-rules).
- `pki/`: given a CA bundle, signing produces a cert that parses, has the expected CN, networks, groups, validity window. Round-trip through `cert.UnmarshalNebulaCertificateFromPEM`.
- `manifest/`: round-trip JSON, schema_version pinning, missing-fields handling.
- `plan/`: given (config, fake fs, fake manifest), assert the action list. This is where idempotency lives — same inputs twice → second plan is empty.
- `fsutil/`: atomic write semantics (temp file + rename), parent-dir creation.

### End-to-end tests via `testscript`

Picked over alternatives (plain Go integration, Bats, godog) because it expresses exactly what we need: invoke the binary, assert stdout/stderr/exit code, and assert the resulting **file tree** — including running the binary twice and `cmp`-ing the tree to prove idempotency.

Layout:

```
test/e2e/
  testdata/
    script/
      generate-ca.txtar
      reference-ca.txtar
      output-dir.txtar
      idempotent-rerun.txtar
      multi-ca.txtar
      trust-bundle.txtar
      ca-rotation.txtar
      renew-before.txtar
      in-pub.txtar
      check-valid.txtar
      check-invalid-duplicate-host.txtar
      dry-run-no-writes.txtar
      version.txtar
```

Each script:

1. Writes a `nebula.hcl` (and optional pre-existing CA files for reference-mode scenarios).
2. Runs `nebula-pki [args...]`.
3. Asserts exit code, stdout/stderr patterns.
4. Asserts file existence + (where deterministic) file content via `cmp` against golden files.
5. For idempotency: runs the binary a second time, snapshots the tree, asserts no diff.

Cert and key bytes are non-deterministic (random key material, timestamps in cert validity). Strategy:

- Inject `pki.Clock` (interface) so issuance time is controllable; tests pass a fixed time.
- Cert/key file **existence and parseability** is asserted in the txtar; **structured assertions** on cert internals (CN, groups, networks, fingerprint stability across reruns) live in Go unit tests, not in txtar.
- Manifest JSON is asserted via `cmp` against a golden file, with non-deterministic fields (fingerprint) either pinned through a seeded key source in tests or replaced via a small "redact" helper before compare.

A "verifier" sub-helper compiled into the test binary lets txtar scripts call e.g. `nebula-pki-verify cert out/ca/ca.crt --cn=my-mesh` for structured checks without leaving the script.

### What we deliberately don't test

- The upstream `slackhq/nebula/cert` library itself.
- Real network or SSH (the tool does neither; see [non-goals](../readme.md#non-goals)).
- Cross-platform filesystem edge cases beyond what `t.TempDir()` covers on Linux + macOS.

## Shipping

Both Homebrew and Nix consume **GoReleaser** output. One source of truth for the build.

### GoReleaser

- Cross-compile matrix: `linux/amd64`, `linux/arm64`, `darwin/amd64`, `darwin/arm64`.
- Inject `internal/buildinfo` vars (`Version`, `Commit`, `Date`) via `-ldflags`.
- Archive: `nebula-pki_<version>_<os>_<arch>.tar.gz` with the binary + readme + license.
- Generate Homebrew formula and commit it back into **this** repo at `Formula/nebula-pki.rb` on tagged release (no separate tap repo — single-repo policy).
- Generate a SHA256 checksum file and a source tarball — both consumed by the flake.
- Configured in `.goreleaser.yaml` at repo root.

### Homebrew

- Formula lives **in this repo** at `Formula/nebula-pki.rb`. No separate tap repository.
- GitHub auto-discovers `Formula/*.rb` in any public repo, so `brew tap anverse/nebula-pki` works against the main repo directly. Install path for users: `brew install anverse/nebula-pki/nebula-pki`.
- GoReleaser's `brews:` block is configured with `repository.name: nebula-pki` (this repo) and `directory: Formula`. The release job commits the regenerated formula back to `main` using the workflow-issued `GITHUB_TOKEN` (`permissions: contents: write`). No PAT and no bot account: `main` is unprotected today, and the repo (managed via Terraform) has no branch-protection rules, so the default token suffices.
- If branch protection is ever introduced via Terraform, the rule must explicitly allow the `github-actions[bot]` actor to push to `main`, or the release job will need to be changed to open a PR instead of pushing directly. Either way, no human PAT.
- Smoke-tested in CI by `brew install --build-from-source ./Formula/nebula-pki.rb` once per release.

### Nix flake

- Existing `flake.nix` already provides `devShells.default`. Extend with:
  - `packages.default` — a `buildGoModule` derivation that builds the same binary, with `ldflags` mirroring GoReleaser, and a fixed `vendorHash`.
  - `apps.default` — wired so `nix run github:anverse/nebula-pki` works.
  - `checks.default` — runs `go test ./...` under `nix flake check`.
- `flake.lock` updated and committed alongside dependency bumps.
- Each release tag is cut by `task release VERSION=vX.Y.Z`, which bumps `flake.nix` (`version`, `vendorHash`), commits, tags, and pushes commit + tag atomically *before* the GoReleaser run. The tag therefore points at a tree where `flake.nix` already names the released version, so `nix run github:anverse/nebula-pki/vX.Y.Z -- --version` reports `X.Y.Z`. Mechanism, alternatives, and rationale: [ADR-014](../adr/014-flake-version-sync.md).

### Versioning policy for the pre-v0.1 series

- Tag every shipped iteration as `v0.0.N` (annotated tag).
- `internal/buildinfo.Version` defaults to `"dev"` on unversioned builds; release builds inject the tag.
- `nebula-pki --version` (and `nebula-pki version`) print `version commit date`.
- Pre-1.0 semantics: any v0.0.x → v0.0.(x+1) bump may break the HCL schema. Each breaking change is noted in `CHANGELOG.md` and an [ADR](../adr/) if non-obvious. Final v0.1.0 cut freezes the HCL surface for the lifetime of the 0.1.x line.

### Taskfile

Top-level `taskfile.yml` (already in dev-shell PATH) wraps everything. Targets, with concrete responsibilities:

| task | purpose |
| --- | --- |
| `build` | local `go build -o bin/nebula-pki ./cmd/nebula-pki` with dev ldflags |
| `test` | `go test ./...` |
| `test:e2e` | builds the binary then runs the testscript suite |
| `lint` | `go vet ./...` + `gofmt -l` check |
| `release:snapshot` | `goreleaser release --snapshot --clean` (no publish; smoke-tests the release pipeline locally) |
| `release VERSION=vX.Y.Z` | cut a real release: bumps `flake.nix`, commits, tags, pushes (see [ADR-014](../adr/014-flake-version-sync.md)). The tag-push then triggers GoReleaser in CI. |
| `nix:build` | `nix build .#default` |
| `nix:run` | `nix run .#default -- --version` |

CI runs `lint`, `test`, `test:e2e`, `release:snapshot`, and `nix:build` on every PR. Tag pushes additionally run `release` and the Homebrew tap update.

## Iteration plan: v0.0.1 → v0.1.0

Each step ends with a tagged, installable release. Every step adds tests for its own surface and re-runs every prior test.

### v0.0.1 — skeleton + shipping pipeline

Goal: prove the shipping story end-to-end with a binary that does almost nothing.

- `cmd/nebula-pki` prints version when invoked with `--version` or `version`.
- `internal/buildinfo` populated via ldflags.
- `taskfile.yml` with `build`, `test`, `lint`, `nix:build`, `release:snapshot`.
- `.goreleaser.yaml` configured for the four-arch matrix.
- `flake.nix` extended with `packages.default` + `apps.default`.
- `Formula/` directory in this repo with a stub `nebula-pki.rb` that GoReleaser will overwrite on first release.
- CI: GitHub Actions workflows for PR (`test`, `lint`, `release:snapshot`, `nix build`) and tag (`release`).
- Tests: one unit test on buildinfo, one testscript scenario asserting `nebula-pki --version` output matches the ldflag-injected version, one CI job verifying `nix run` works on Linux.

Done when `brew install anverse/nebula-pki/nebula-pki && nebula-pki --version` works on a clean macOS, and `nix run github:anverse/nebula-pki -- --version` works on Linux.

### v0.0.2 — config parsing + `check`

- `internal/config`: parse the full v0.1 HCL surface (CA generate + reference, host, storage minus encryption, `output_dir`). Decode-only; no I/O.
- All [validation rules](../hcl-schema.md#validation-rules) from the spec implemented and unit-tested.
- `nebula-pki check` subcommand: parse + validate, exit 0 / exit 1 with readable error messages.
- `storage.encryption "<backend>" {}` blocks are parsed but rejected with a clear "encryption ships in a later release" error so users hit a wall, not silent acceptance.
- Tests: exhaustive table-driven config tests; testscript scenarios for `check` happy path and each validation rule.

### v0.0.3 — CA generate mode

- `internal/pki`: CA generation wrapping `slackhq/nebula/cert`; pure (clock-injectable).
- `internal/plan` + `internal/apply` MVP: only handles the CA action.
- `internal/manifest`: read/write `nebula-pki.json` with `schema_version = 1` and `ca` block.
- Default `nebula-pki` (no args) reconciles CA only; hosts ignored for now but parsed.
- Tests: unit tests for cert structure (CN, validity, curve), manifest round-trip, testscript for first-run (writes CA + manifest) and second-run (no churn, no diff).

### v0.0.4 — CA reference mode

- `ca { cert_file, key_file }` path: load existing CA, populate manifest in `mode: "reference"`, never rewrite the files.
- Validation: reject generate-only fields in reference mode (already in `config`, now enforced end-to-end).
- Tests: testscript scenario pre-seeds `ca.crt`/`ca.key`, asserts they're untouched after run.

### v0.0.5 — host signing (basic)

- Host certs signed under the loaded/generated CA, written to default `out/hosts/<name>.crt` / `.key`.
- Manifest `hosts` map populated per ADR-002.
- Tests: cert structure unit tests (CN = host.name, groups, networks, IP subset of CA's), idempotency scenario (re-run = no diff), validation scenarios (host groups not in CA groups → error).

### v0.0.6 — custom output directory per host

- `host.output_dir` — optional per-host destination directory. Cert/key written as `<dir>/<host.name>.crt` and `<dir>/<host.name>.key`. Defaults to `<storage.out_dir>/hosts`.
- `out_crt` / `out_key` — redefined as path components joined onto `output_dir` when that is set; otherwise resolved relative to the config file. No longer mutually exclusive with `output_dir`; they compose.
- When the resolved output path changes between runs (any of `output_dir`, `out_crt`, `out_key`), the host is re-signed at the new location; the CLI emits a notice for any old path that still exists on disk.
- Manifest `hosts.<key>.artifacts` always exactly one entry; `dir` field set to `output_dir` when configured.
- Tests: testscript scenario with custom `output_dir`; dir-change scenario asserting re-sign + stale notice; `out_crt` with `output_dir` combination.

### v0.0.7 — `--dry-run` + polish

- `--dry-run` builds the plan and prints it as a human-readable diff (`+ write out/ca/ca.crt`, etc.), no writes.
- Stderr/stdout cleaned up: progress on stderr, machine-readable output (if any) on stdout.
- `nebula-pki --help` / per-command help reviewed.
- Tests: dry-run testscript asserts zero file mutations regardless of config state.

### v0.0.8 — multi-CA substrate

Goal: two CAs can coexist in one file and sign different hosts. No rotation semantics yet — just selection. This is the schema-shape change, landed before the freeze. See [ADR-015](../adr/015-multiple-cas-per-config.md).

- `internal/config`: parse labelled `ca "<label>" {}` blocks alongside the legacy unlabelled `ca {}`; parse `ca.default` and `host.ca`. Reject mixing labelled + unlabelled; reject `host.ca`/`ca.default` in single-CA files; reject more than one `default = true`; reject undeclared CA references; reject ambiguous signing CA.
- Per-CA restriction scoping: `host.groups`/`networks`/`unsafe_networks` validated against the host's signing CA.
- `internal/pki` + `plan` + `apply`: generate/reference each CA; sign each host under its resolved CA; default CA cert/key paths become `ca/<label>.crt` in the multi-CA shape.
- `internal/manifest`: `cas` map keyed by label (back-compat `ca` retained for single-CA), per-host `ca` label, per-CA `default` marker.
- Tests: config tests for every new validation rule; signing tests asserting the right CA signed each host (`ca_fingerprint`); manifest round-trip for `cas`; idempotency with two CAs.

### v0.0.9 — trust bundle

Goal: emit the `pki.ca` bundle. Useful even with one CA (stable downstream path). See [ADR-016](../adr/016-ca-rotation-and-trust-bundles.md).

- `apply`: write `out/ca/bundle.crt` (renameable via `storage.trust_bundle_file`) — concatenated PEM of every active (non-`archived`) CA cert, in a deterministic order, written atomically.
- `internal/manifest`: `trust_bundle` record (`path`, `ca_fingerprints`).
- Bundle is part of the idempotency comparison: an unchanged active-CA set leaves `bundle.crt` byte-identical.
- Tests: single-CA bundle equals the lone CA cert; two-CA bundle concatenates both; testscript asserts byte-stable bundle across reruns and that `ca_fingerprints` matches.

### v0.0.10 — CA rotation + renewal

Goal: the upstream four-step rotation becomes `edit HCL + re-run`, and hosts re-sign before expiry. See [ADR-016](../adr/016-ca-rotation-and-trust-bundles.md) and [ADR-017](../adr/017-host-renewal-threshold.md).

- `ca.archived` excludes a CA from the bundle and bars it from signing (validation + apply).
- Moving the `default = true` marker between CAs (or changing a per-host `ca`) re-signs affected hosts — already implied by the `ca_fingerprint` clause of the idempotency rule; add explicit coverage.
- `renew_before` on host and CA: extend the idempotency verdict with clause 7 (`now + renew_before < not_after`); resolve host-over-CA; validate `renew_before < effective validity`.
- Post-run deadline reporting: after reconcile and `--dry-run` (including no-op runs), print to stderr the earliest actionable deadline — the soonest of (a host entering its `renew_before` window) and (the expiry of any cert with no threshold) — plus an "expiring soon" summary and a "run again before <date>" hint. Advisory only; no exit-code or write effect. See [ADR-017](../adr/017-host-renewal-threshold.md).
- `internal/manifest`: per-host resolved `renew_before`.
- Tests (clock-injected): full rotation scenario (add `next` → bundle has both → move `default = true` to `next` → hosts re-signed under `next` → archive `current` → bundle drops it); renewal scenario (host inside window re-signs once, then is byte-identical; outside window writes nothing); validation for archived-CA-signing, two-defaults, and `renew_before >= validity`; deadline-line output asserts the earliest deadline and the no-op-run-still-prints behaviour.

### v0.0.11 — `in_pub` air-gapped signing

Goal: sign device-supplied public keys with no private key on the CA host. Config-only; no `keygen` subcommand. See [ADR-018](../adr/018-in-pub-air-gapped-signing.md).

- `apply`/`pki`: when `host.in_pub` is set, read the PEM public key, verify its curve matches the signing CA, sign and write **only** the cert (fanned out), write no key, apply no encryption.
- Validation: reject `in_pub` + `out_key`; curve-mismatch error at reconcile/`check` (file must be read).
- `internal/manifest`: per-host `in_pub: true`; `artifacts` entries omit `key_path`.
- Renewal of an `in_pub` host re-signs the same public key.
- Tests: sign a fixture `.pub` (generated via the `cert` library in test), assert cert-only output and manifest provenance; curve-mismatch failure; `in_pub`+`out_key` validation error; renewal re-signs same key.

### v0.1.0 — cut

- Final pass over readme.md and agents.md so examples match the shipped behaviour (multi-CA, rotation, trust bundle, `renew_before`, `in_pub`).
- `task release VERSION=v0.1.0` — bumps the flake, tags, pushes; CI does the rest (GoReleaser, GitHub release, Homebrew formula commit). See [ADR-014](../adr/014-flake-version-sync.md).

## Out of scope for the 0.1 line (still)

- `storage.encryption` of any kind. First target for v0.2.
- `nebula-pki show`. Operators `jq` the manifest until then.
- Anything in the spec's [deferred capabilities](../readme.md#deferred-capabilities).
