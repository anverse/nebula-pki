# Milestone v0.2 — encryption, CA cert distribution, and reencrypt subcommand

End state: an operator can configure `encryption "sops" {}` and have every private key written sops-encrypted (via `.sops.yaml` discovery or inline recipients); use `encryption "external" {}` to plug in any command-line encryption tool; run `nebula-pki reencrypt` to re-encrypt all key files when recipients change; and declare `link_crt` on a `ca` block to place git-committable symlinks of the CA certificate into any host output directory that needs it.

Versioning: iterations between v0.1.0 and the v0.2 cut ship as `v0.1.x` tags. Each is a real, installable release.

The encryption backend shells out to the `sops` CLI rather than embedding the `getsops/sops` Go library. This keeps the binary lean (no transitive AWS/GCP/Azure SDKs) and defers sops release tracking to the operator. See [ADR-003](../adr/003-encryption-strategy.md).

## Acceptance criteria for v0.2.0

### SOPS encryption (`encryption "sops"`)

- All CA private key files (generate mode) and host private key files are written sops-encrypted with the configured recipients.
- Empty block (`encryption "sops" {}`) — the `sops` binary performs its standard upward search for `.sops.yaml` from the temp file's directory (placed in the same directory as the output file); no inline recipients required.
- Inline recipients (`age`, `pgp`, `kms`, `gcp_kms`, `azure_kv`, `hc_vault_transit`) take precedence over `.sops.yaml` for files written by nebula-pki — same precedence rules as `sops --encrypt --age ... --pgp ...`.
- `shamir_threshold` and `config` (explicit `.sops.yaml` path) are honoured when set.
- `output_suffix` (default `".enc"`) appended to key file names on disk; the manifest records paths **with** the suffix so it exactly describes what is on disk.
- Only **private key files** are encrypted. `.crt` certificates, QR PNGs, the manifest, and the trust bundle are never encrypted and never suffixed.
- **Decryption on reuse** — when a CA key was previously written as `.key.enc`, the tool decrypts it in-memory by shelling out to `sops --decrypt --input-type binary --output-type binary`; no plaintext temp file is written to disk. The `sops` binary must be in PATH for any reconcile that signs hosts under an encrypted CA key.
- **Idempotency** — a re-run against an unchanged config produces no key-file writes; existing encrypted key files are left untouched.
- **Recipient-mismatch warning** — when the `recipients_sha` stored in the manifest for an existing encrypted key file differs from the hash of the currently configured recipients, the CLI prints a warning on every run until `nebula-pki reencrypt` re-encrypts the file. Applies only when both sides have a non-empty hash (i.e., inline recipients were configured for both the stored and current run); `.sops.yaml`-only mode emits no mismatch warning.

### External encryption backend (`encryption "external"`)

- `encryption "external" { encrypt_command = [...], decrypt_command = [...] }` — invokes operator-supplied commands to encrypt and decrypt key files. `{{.InPath}}` / `{{.OutPath}}` are substituted with absolute temp-file paths at invocation time; see ADR-023 for the full protocol.
- `output_suffix` behaves identically to the sops backend.
- Decryption on reuse: when a CA key exists at a suffixed path, the `decrypt_command` is invoked; decrypted bytes are piped through stdout into memory, no plaintext temp file.
- Recipient-mismatch warning: when a SHA-256 of the `encrypt_command` slice stored in the manifest differs from the current config's hash, warn on every run until `nebula-pki reencrypt` is executed.

### Reencrypt subcommand (`nebula-pki reencrypt`)

- `nebula-pki reencrypt` — reads every managed key file that the manifest records as encrypted, decrypts it using the backend and credentials that produced it, and re-encrypts it with the current backend and recipients. Atomic writes (temp-then-rename); aborts on first error leaving previously written files intact.
- `--dry-run` flag supported (prints what would be re-encrypted, no writes).
- `--force` flag — re-encrypts all managed key files regardless of whether a mismatch is detected. Intended for external-backend cases where the encryption behaviour changed without the tracked command string changing.
- After a successful `reencrypt`, the mismatch warning is cleared (manifest `recipients_sha` updated).
- Works across both `sops` and `external` backends (old and new backend may differ, enabling migration between backends).

### CA cert links (`link_crt`)

- `ca "<label>" { link_crt = ["dir/a", "dir/b", ...] }` — creates a symlink at `<dir>/<symlink-name>` in each listed directory, pointing to this CA's certificate file.
- Symlink name follows the CA cert naming: `<label>.crt` by default; if `out_crt` is set, the basename of the resolved `out_crt` path.
- Symlink targets are **relative** paths computed via `filepath.Rel`, so they survive being committed to git and cloned to a different absolute path.
- Parent directories are created if they do not exist.
- **Idempotency** — correct target → no-op; broken or wrong target → recreated; regular file at the path → error (never clobbered).
- **Stale link cleanup** — when a path is removed from `link_crt` between runs, the old symlink is deleted. If a regular file now occupies the path, the tool prints a notice and skips deletion.
- The manifest records each symlink under `cas.<label>.links` as `{"path": "...", "target": "..."}`.

## Decisions

### D-1: What gets encrypted

Only private key files (`.key`, written as `.key<suffix>`) are encrypted. Public certificates (`.crt`), QR PNGs, the trust bundle, and the manifest contain no private material and are left plaintext.

See [ADR-003](../adr/003-encryption-strategy.md#what-gets-encrypted).

### D-2: Sops via CLI subprocess, not Go library

`encryption "sops"` shells out to the `sops` binary rather than embedding `getsops/sops/v3`. This keeps the binary lean (no transitive AWS/GCP/Azure/Vault SDKs), means sops version upgrades are the operator's concern, and is consistent with how the `external` backend works. Operators who choose `encryption "sops"` already have sops installed.

See [ADR-003](../adr/003-encryption-strategy.md).

### D-3: No automatic re-encryption on recipient change

Changing recipient fields between runs does **not** trigger automatic re-encryption. Key files are only (re-)written when the key material itself changes. When the manifest-stored `recipients_sha` differs from the current config's hash, a persistent warning is emitted; the operator resolves it explicitly with `nebula-pki reencrypt`.

See [ADR-003](../adr/003-encryption-strategy.md#recipient-mismatch-warning).

### D-4: Manifest key paths include encryption suffix

The manifest stores artifact key paths **as they exist on disk**, including the encryption suffix (e.g., `out/hosts/app_01.key.enc`). The `encryption.suffix` field in the manifest allows the tool to locate encrypted files even if `output_suffix` is later reconfigured. No schema-version bump is needed.

### D-5: `external` backend ships in v0.2

The `external` backend gives operators an escape hatch for any encryption tool: custom KMS wrappers, standalone `age`, anything. Since the subprocess architecture is already established for the sops backend, the implementation is a thin layer. A good tool delivers its escape hatch by day one.

See [ADR-006](../adr/006-storage-backend-extensibility.md).

### D-6: `link_crt` uses a list of directories; symlink name follows CA cert filename

`link_crt` is a list of **directories**. The symlink filename is derived from the CA cert filename (`<label>.crt` or `out_crt` basename), reusing existing path logic.

See [ADR-021](../adr/021-ca-cert-links.md).

### D-7: Symlinks are always relative

Symlink targets are computed as relative paths via `filepath.Rel(linkDir, caAbsCertPath)`. Absolute symlinks break after `git clone` to a different path.

See [ADR-021](../adr/021-ca-cert-links.md).

## Iteration plan: v0.1.0 → v0.2.0

Each step ships as `v0.1.x`. Every step re-runs the full prior test suite.

### v0.1.1 — sops write path

- `internal/crypto/`: `Encryptor` interface, `NoneBackend`, `SopsBackend.Encrypt` (CLI subprocess).
- Config: `encryption "sops"` / `encryption "none"` decoded; wired through to `apply`.
- `plan`: `planCA`/`planHost` check existence against `path + suffix`; set `EncryptKey` flag.
- `apply`: `writeKeyFile()` helper; manifest records encrypted key paths with `EncryptionRecord`.
- `caPEMs.encrypted` guard: signing hosts under an encrypted CA key on the same run (all noop) returns a clear "requires v0.1.2" error.
- Tests: `sops-age-encrypt.txtar`, `sops-dotfile-discovery.txtar`, `sops-idempotent.txtar`.

### v0.1.2 — sops decrypt path and mismatch warning

- `SopsBackend.Decrypt`: invoke `sops --decrypt --input-type binary --output-type binary <tmpfile>`; pipe stdout to memory; defer temp file removal.
- Add `Decryptor` interface to `internal/crypto/backend.go`; `SopsBackend` implements it.
- `reconcileOneCA(OpNoop, EncryptKey=true)`: decrypt in-memory; remove the `caPEMs.encrypted` guard; host signing on rerun works.
- Mismatch check: on plan build, for each manifest-tracked encrypted artifact compare `manifest.encryption.recipients_sha` vs. `enc.RecipientsHash()`; accumulate and print warnings after plan.
- Tests: `sops-ca-key-decrypt.txtar`, `sops-mismatch-warning.txtar`; decrypt round-trip unit test.

### v0.1.3 — `link_crt` symlinks

- Config: `link_crt []string` on `ca` block; validation in `config.Validate`.
- `plan`: `CreateSymlink` / `DeleteSymlink` actions from diff of declared vs. manifest links.
- `apply`: relative-target computation via `filepath.Rel`; parent mkdir; `os.Lstat` guard before any remove.
- Manifest: `cas.<label>.links` array.
- Tests: all `link-crt-*.txtar` scenarios.

### v0.1.4 — `external` encryption backend

- `internal/crypto/ExternalBackend`: `Encrypt` and `Decrypt` subprocess; `{{.InPath}}`/`{{.OutPath}}` substitution with stdin/stdout fallback; mismatch detection via SHA-256 of `encrypt_command` slice. Both commands required. See ADR-023.
- Config: `encryption "external"` decoded into `ExternalConfig`.
- Tests: `external-encrypt-decrypt.txtar`, `external-mismatch-warning.txtar`.

### v0.1.5 — `nebula-pki reencrypt`

- `cmd/nebula-pki/reencrypt.go`: cobra subcommand; iterate manifest artifacts; reconstruct old backend from `encryption.backend` + stored hash; decrypt with current credentials; re-encrypt with current config; write atomically; update manifest.
- `--dry-run` and `--force` flags.
- Tests: `reencrypt-sops.txtar`, `reencrypt-dry-run.txtar`, `reencrypt-backend-migration.txtar`, `reencrypt-force.txtar`.

### v0.2.0 — cut

- Final pass: `readme.md`, `spec/hcl-schema.md` examples show `.key.enc` names, symlinks in artifact tree, `nebula-pki reencrypt` in CLI reference.
- Verify mismatch warning language is clear and actionable in all three backends.
- `task release VERSION=v0.2.0`.

## Out of scope for the 0.2 line

- `nebula-pki show` — operators `jq` the manifest until then.
- PKCS#11 / HSM-backed CAs.
- Automatic `link_crt` population from declared `host.output_dir` values.
- Anything in the spec's [deferred capabilities](../readme.md#deferred-capabilities).
