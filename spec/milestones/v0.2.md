# Milestone v0.2 — encryption and CA cert distribution

End state: an operator can configure `encryption "sops" {}` and have every private key written sops-encrypted (via `.sops.yaml` discovery or inline recipients), and can declare `link_crt` on a `ca` block to place git-committable symlinks of the CA certificate into any directory that needs it — including each host's `output_dir`.

Versioning: iterations between v0.1.0 and the v0.2 cut ship as `v0.1.x` tags. Each is a real, installable release.

---

## Acceptance criteria for v0.2.0

### SOPS encryption

- `storage { encryption "sops" { ... } }` — all host private key files and CA private key files (generate mode) are written sops-encrypted with the configured recipients.
- Empty block (`encryption "sops" {}`) — the sops library performs its standard upward search for `.sops.yaml` from each output file's directory; no inline recipients required.
- Inline recipients (`age`, `pgp`, `kms`, `gcp_kms`, `azure_kv`, `hc_vault_transit`) take precedence over `.sops.yaml` for files written by nebula-pki — same precedence as passing flags on the sops CLI.
- `shamir_threshold` and `config` (explicit `.sops.yaml` path) are honoured when set.
- `output_suffix` (default `".enc"`) appended to key file names; the manifest records paths with the suffix so they describe exactly what is on disk.
- Only private key files are encrypted. Public certificates, QR PNGs, the manifest, and the trust bundle are never encrypted and never suffixed.
- **Decryption on subsequent runs** — when a CA key was previously written as `.key.enc` (generate mode), the tool decrypts it in-process before using it to sign hosts; no external `sops` binary is required.
- **Idempotency** — a re-run against an unchanged config produces no writes; existing encrypted key files are left untouched.
- `encryption "external" {}` remains parse-accepted and returns a clear "not yet implemented" error (deferred beyond v0.2).

### CA cert links (`link_crt`)

- `ca "<label>" { link_crt = ["path/to/symlink.crt", ...] }` — the tool creates a symlink at each listed path pointing to this CA's certificate file.
- Symlink targets are **relative** paths (computed from the symlink location to the CA cert using `filepath.Rel`), so they survive being committed to git and cloned to a different absolute prefix.
- The parent directory of each symlink path is created if it does not exist (consistent with `output_dir` behaviour).
- **Idempotency** — on re-run, each symlink is verified: if it exists and points to the correct target, no action is taken; if it is broken or points elsewhere, it is recreated; if a regular file (not a symlink) already exists at the path, the run fails with an error rather than clobbering.
- **Stale link cleanup** — if a path is removed from `link_crt` between runs, the old symlink is deleted (the manifest records which links were last declared; removals are detected by diffing manifest vs. config).
- The manifest records each symlink under `cas.<label>.links` as `{"path": "...", "target": "..."}` (symlink path + resolved relative target).
- `schema_version` in the manifest bumps to `2` in this milestone (additive; v1 manifests are still readable and are migrated on first v0.2 run).

## Open questions and decisions needed

The following items are not yet resolved. They should each be closed — either as an ADR or as an inline note in `hcl-schema.md` — before coding the relevant step.

### SOPS

**OQ-1: `external` backend scope**
The spec documents `encryption "external" { encrypt_command, decrypt_command }`. Is this in scope for v0.2 or does it remain a parse-and-error stub?
Recommendation: keep it stubbed; `sops` is the priority and `external` is its own implementation effort.

**OQ-2: Recipient-change / rekey behaviour**
If an operator changes the `age` or `pgp` recipient list in the HCL between runs, should existing `.key.enc` files be re-encrypted with the new recipients?
- Option A: Yes — detect the mismatch by comparing sops metadata of the existing file against configured recipients; re-encrypt automatically.
- Option B: No — only new or changed key material triggers a write; rekey is a separate explicit operation.
Recommendation: **Option B** for v0.2 (simpler, avoids decrypting and re-encrypting valid files on every config edit). A future `nebula-pki rekey` subcommand can address this.

**OQ-3: Decryption credential availability**
When a CA key is encrypted, the operator's decryption credential (age private key, PGP keyring, AWS creds, …) must be present on the workstation at every reconcile run that involves signing hosts — not just at generation time. This is a real operational constraint that should be documented explicitly in `hcl-schema.md` under the `encryption "sops"` block (and noted in the operator guide / readme). No code decision required, but the documentation gap must be closed before v0.2.0.

**OQ-4: Manifest key-path format**
The manifest currently stores artifact paths without an encryption suffix. With sops enabled, the on-disk name changes (`host.key` → `host.key.enc`). The manifest should store the **on-disk path** (with suffix) so it exactly describes what is present on disk. Readers that need the "logical" key path strip the suffix themselves. This is a breaking change to the manifest shape → motivates the `schema_version` 2 bump.

### CA cert links

**OQ-5: Field semantics — list of paths vs. list of directories**

Two designs are on the table:

*Option A — list of explicit symlink paths (recommended)*
```hcl
ca "mesh" {
  link_crt = ["out/hetzner/ca.crt", "out/aws/ca.crt"]
}
```
The operator names each symlink file. Allows using a conventional `ca.crt` name in each host directory regardless of the CA label. Unambiguous in the manifest (the path is the symlink path).

*Option B — list of directories*
```hcl
ca "mesh" {
  link_crt = ["out/hetzner", "out/aws"]
}
```
The symlink name is derived from the CA cert filename (`mesh.crt`). Less typing; less control over the symlink name.

**Decision needed.** Recommendation: **Option A** — explicit paths give the operator control over the symlink name, which matters because Nebula configs typically reference `ca.crt` by a stable conventional name, not by the CA label. The verbosity is minor and the explicitness reduces surprises.

**OQ-6: Symlink target computation**
The relative target is computed as `filepath.Rel(dir(linkPath), caAbsCertPath)`. Two edge cases:

- Symlink and CA cert are on different filesystems (or the relative path would cross a filesystem boundary): relative path is impossible. Tool should error with a clear message suggesting the operator use an absolute symlink instead — but in v0.2, relative is the only supported mode (an `--abs-links` flag can be added if a real need arises).
- The computed relative path escapes the git repository root (e.g., `../../../../foo/ca.crt`): allowed, but the tool should print a warning since such a symlink may break after a fresh clone.

**OQ-7: Interaction with per-host `output_dir`**
The primary motivation for `link_crt` is to place the CA cert alongside host artifacts that live in a custom `output_dir`. The operator must keep the `link_crt` list in sync with their `output_dir` values manually — the tool does not auto-populate `link_crt` from declared `output_dir` values. This is intentional (explicit over magic), but should be called out clearly in `hcl-schema.md` with an example that pairs a `host.output_dir` value with its corresponding `link_crt` entry on the CA.

**OQ-8: Stale link deletion safety**
When a path is removed from `link_crt`, the tool deletes the symlink on next run (if the manifest records it as previously managed). It does **not** delete regular files at that path — only symlinks. If the path has been replaced by a real file in the meantime, the tool prints a notice and skips deletion. This mirrors the "never auto-delete cert or key files" policy from ADR-020 extended to symlinks-that-have-become-files.

## Code changes

### `internal/crypto/` (new package)

Interface and implementations for encryption/decryption:

```
internal/crypto/
  backend.go          # Encryptor / Decryptor interfaces
  none.go             # NoneBackend: pass-through (identity)
  sops.go             # SopsBackend: wraps getsops/sops in-process
```

`Encryptor.Encrypt(plaintext []byte, outputPath string) ([]byte, error)` — encrypts the payload; uses `outputPath` for `.sops.yaml` creation-rule matching when no inline recipients are configured.

`Decryptor.Decrypt(ciphertext []byte, inputPath string) ([]byte, error)` — decrypts using whichever key material is available in the operator's environment.

`NoneBackend` is a pass-through (returns plaintext unchanged). Used when `encryption "none" {}` is active or when `in_pub` signing produces no key.

### `internal/config/`

- Remove the parse-then-error stub for `encryption "sops"`. Parse into `SopsConfig` struct and surface it in the resolved config.
- Add `LinkCrt []string` field to the `CA` struct (HCL attribute `link_crt`). Validation: non-empty list entries, no duplicates, each entry is a non-empty string. Relative paths resolve against the config file's directory.

### `internal/plan/`

- New action `WriteEncryptedFile` (emitted instead of `WriteFile` for key artifacts when backend is not `none`). Carries the plaintext content; `apply` handles encryption.
- CA key decryption injected into the signing pre-step: when the resolved CA key path has the active encryption suffix, decrypt in-memory before passing to `pki.SignHost`. No temporary file on disk.
- New action `CreateSymlink { LinkPath, Target string }` — emitted when a declared link is absent, broken, or stale.
- New action `DeleteSymlink { LinkPath string }` — emitted when a manifest-recorded link is no longer declared in the config.

### `internal/apply/`

- `WriteEncryptedFile`: calls `Encryptor.Encrypt`, writes ciphertext atomically (temp + rename, same as `WriteFile`).
- `CreateSymlink`: calls `os.Symlink`; creates parent dir if needed.
- `DeleteSymlink`: calls `os.Remove` only if the path is confirmed to be a symlink (`os.Lstat` + `mode.Type() == fs.ModeSymlink`); skips with a notice if it is a regular file.

### `internal/manifest/`

- CA record gains `"links": [{"path": "...", "target": "..."}]`.
- Artifact key paths stored with encryption suffix when backend is active.
- `schema_version` 2 written on any v0.2 run; v1 manifests are read and treated as if `links` is absent and key paths have no suffix (effectively migrated in-memory on first v0.2 run, then written as v2).

## Iteration plan: v0.1.0 → v0.2.0

Each step ships as `v0.1.x`. Every step re-runs the full prior test suite.

### v0.1.1 — sops write path

- Add `getsops/sops` to `go.mod`.
- `internal/crypto/`: `Encryptor` interface, `NoneBackend`, `SopsBackend.Encrypt`.
- Remove the parse-then-error stub for `encryption "sops"`; wire the parsed config through to `apply`.
- `plan`: key-file actions emit `WriteEncryptedFile` when backend is sops; cert/QR/manifest actions remain `WriteFile`.
- `output_suffix` honoured; manifest paths stored with suffix.
- `schema_version` bumped to 2 (covers suffix-in-paths + future links field).
- Tests: `sops-age-encrypt.txtar`, `sops-dotfile-discovery.txtar`, `sops-idempotent.txtar`, manifest v2 round-trip.

### v0.1.2 — sops read/decrypt path

- `SopsBackend.Decrypt`.
- `plan`: when a CA key path carries the encryption suffix, decrypt in-memory before handing to `pki`; no plaintext temp file written to disk.
- Tests: `sops-ca-key-decrypt.txtar`; existing CA rotation tests pass with an encrypted CA key.

### v0.1.3 — `link_crt` symlinks

- Resolve OQ-5 (path vs. directory); write ADR-021.
- Config parsing + validation for `link_crt` on the `ca` block.
- `CreateSymlink` / `DeleteSymlink` actions in `plan` and `apply`.
- Manifest: `cas.<label>.links` array (schema_version is already 2 from v0.1.1).
- Tests: all five `link-crt-*.txtar` scenarios.

### v0.2.0 — cut

- Final pass: `readme.md`, `agents.md`, `spec/hcl-schema.md` examples updated to show `.key.enc` names and symlinks in the artifact tree.
- Close any open documentation gaps identified under OQ-3 (credential constraint language).
- `task release VERSION=v0.2.0`.
