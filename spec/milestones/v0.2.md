# Milestone v0.2 â€” encryption, CA cert distribution, and rekey

End state: an operator can configure `encryption "sops" {}` and have every private key written sops-encrypted (via `.sops.yaml` discovery or inline recipients); use `encryption "external" {}` to plug in any command-line encryption tool; run `nebula-pki rekey` to re-encrypt all key files when recipients change; and declare `link_crt` on a `ca` block to place git-committable symlinks of the CA certificate into any host output directory that needs it.

Versioning: iterations between v0.1.0 and the v0.2 cut ship as `v0.1.x` tags. Each is a real, installable release.

## Acceptance criteria for v0.2.0

### SOPS encryption (`encryption "sops"`)

- All CA private key files (generate mode) and host private key files are written sops-encrypted with the configured recipients.
- Empty block (`encryption "sops" {}`) â€” the sops library performs its standard upward search for `.sops.yaml` from each output file's directory; no inline recipients required.
- Inline recipients (`age`, `pgp`, `kms`, `gcp_kms`, `azure_kv`, `hc_vault_transit`) take precedence over `.sops.yaml` for files written by nebula-pki.
- `shamir_threshold` and `config` (explicit `.sops.yaml` path) are honoured when set.
- `output_suffix` (default `".enc"`) appended to key file names on disk; the manifest records paths **with** the suffix so it exactly describes what is on disk.
- Only **private key files** are encrypted. `.crt` certificates, QR PNGs, the manifest, and the trust bundle are never encrypted and never suffixed. See [ADR-003](./adr/003-encryption-strategy.md#what-gets-encrypted).
- **Decryption on reuse** â€” when a CA key was previously written as `.key.enc`, the tool decrypts it in-process (no external `sops` binary required) before using it to sign hosts.
- **Idempotency** â€” a re-run against an unchanged config produces no key-file writes; existing encrypted key files are left untouched.
- **Persistent mismatch warning** â€” when the sops metadata of an existing encrypted key file does not match the currently configured recipients, the CLI prints a warning on every run until `nebula-pki rekey` is executed. See [ADR-003](./adr/003-encryption-strategy.md#recipient-mismatch-warning).

### External encryption backend (`encryption "external"`)

- `encryption "external" { encrypt_command = [...], decrypt_command = [...] }` â€” invokes operator-supplied commands to encrypt and decrypt key files. `{{.In}}` / `{{.Out}}` are substituted with absolute paths at invocation time.
- `output_suffix` behaves identically to the sops backend.
- Decryption on reuse: when a CA key exists at a suffixed path, the `decrypt_command` is invoked before signing.
- Persistent mismatch warning: when the `encrypt_command` stored in the manifest differs from the currently configured one, warn on every run until `nebula-pki rekey` is executed.

### Rekey subcommand (`nebula-pki rekey`)

- `nebula-pki rekey` â€” reads every managed key file that the manifest records as encrypted, decrypts it using the backend and credentials that produced it, and re-encrypts it with the current backend and recipients.
- `--dry-run` mode supported (prints what would be rekeyed, no writes).
- After a successful rekey, the mismatch warning is cleared.
- Works across both `sops` and `external` backends (the old and new backend may differ, enabling migration between backends).

### CA cert links (`link_crt`)

- `ca "<label>" { link_crt = ["dir/a", "dir/b", ...] }` â€” creates a symlink at `<dir>/<symlink-name>` in each listed directory, pointing to this CA's certificate file. See [ADR-021](./adr/021-ca-cert-links.md).
- Symlink name follows the same label/`out_crt` logic used for the CA cert itself: defaults to `<label>.crt`; if `out_crt` is set, uses the basename of the resolved `out_crt` path.
- Symlink targets are **relative** paths computed from the symlink location to the CA cert file via `filepath.Rel`, so they survive being committed to git and cloned to a different absolute path.
- The parent directory of each symlink is created if it does not exist.
- **Idempotency** â€” existing symlinks are verified: correct target â†’ no-op; broken or wrong target â†’ recreated; regular file at the path â†’ error (never clobbered).
- **Stale link cleanup** â€” when a path is removed from `link_crt` between runs, the old symlink is deleted. If a regular file now occupies that path, the tool prints a notice and skips deletion.
- The manifest records each symlink under `cas.<label>.links` as `{"path": "...", "target": "..."}`.

## Decisions

### D-1: What gets encrypted

Only private key files (`.key`, written as `.key<suffix>`) are encrypted. Public certificates (`.crt`), QR PNGs, the trust bundle, and the manifest contain no private material and are left plaintext.

**Rationale:** In Nebula's design, certificates are public â€” hosts broadcast them during connection establishment. Encrypting certs would add downstream friction (every Terraform `file()` call, every Ansible copy task would need decryption) with zero security benefit. The manifest is explicitly designed to contain no secrets. Operators who want to encrypt the cert tree for metadata reasons (e.g., hiding network topology) should do so at the git-repository or filesystem level, outside this tool's scope.

See [ADR-003](./adr/003-encryption-strategy.md#what-gets-encrypted).

### D-2: Recipient-change / rekey behaviour

Changing the `age`, `pgp`, or other recipient fields between runs does **not** trigger automatic re-encryption of existing key files. Key files are only (re-)written when the key material itself changes (new cert generation or renewal). When recipients differ from what was used to encrypt existing files, a **persistent warning** is emitted on every run until `nebula-pki rekey` clears it.

**Rationale:** Automatically re-encrypting on every config edit would silently rewrite secrets and could produce mixed states in multi-operator environments. Making the rekey explicit lets operators control when credentials rotate and keeps normal runs side-effect-free beyond what the cert state requires. The persistent warning ensures the divergence is not silently ignored.

See [ADR-003](./adr/003-encryption-strategy.md#recipient-mismatch-warning).

### D-3: Manifest key paths include encryption suffix

The manifest stores artifact key paths **as they exist on disk**, including the encryption suffix (e.g., `out/hosts/app_01.key.enc`). Readers that need the logical key name strip the configured `output_suffix`. This requires no `schema_version` bump â€” the tool is not yet in production use and the manifest format can absorb this without a migration story.

### D-4: `external` backend in v0.2

The `external` backend ships in v0.2 (step v0.1.4). Principle: a good tool delivers its escape hatch by day one. The `external` contract (argv + `{{.In}}`/`{{.Out}}` + exit code) is already specified in [ADR-006](./adr/006-storage-backend-extensibility.md); the implementation is straightforward once the encryption interface exists.

### D-5: `link_crt` uses directories; symlink name follows CA cert filename

`link_crt` is a list of **directories**. The symlink filename within each directory is derived the same way as the CA cert filename itself: `<label>.crt` by default, or the basename of the resolved `out_crt` path when `out_crt` is set on the `ca` block. This reuses existing path-resolution logic and keeps the schema surface minimal.

See [ADR-021](./adr/021-ca-cert-links.md).

### D-6: Symlinks are always relative

Symlink targets are always computed as relative paths (via `filepath.Rel(linkDir, caAbsCertPath)`). Absolute symlinks are not supported.

**Rationale:** Committed symlinks must survive `git clone` to a different path. An absolute symlink (`/Users/fw/Projects/.../ca/mesh.crt`) is a hardcoded reference to one machine's filesystem; it breaks on any other checkout location. A relative symlink (e.g., `../../ca/mesh.crt`) is path-agnostic and always valid as long as the directory structure within the repo is preserved â€” which is exactly what git guarantees. There is no `--abs-links` escape hatch: if the two paths are on different filesystems and a relative path cannot be computed, the tool errors with a clear message; in that scenario the operator creates the symlink manually outside the tool's scope.

See [ADR-021](./adr/021-ca-cert-links.md).

### D-7: No automatic `link_crt` population from `output_dir`

The tool does not auto-generate `link_crt` entries from declared `host.output_dir` values. Operators list directories explicitly. See [ADR-021](./adr/021-ca-cert-links.md#interaction-with-output_dir).

## Code changes

### `internal/crypto/` (new package)

```
internal/crypto/
  backend.go       # Encryptor / Decryptor interfaces + BackendConfig
  none.go          # NoneBackend: pass-through identity
  sops.go          # SopsBackend: getsops/sops in-process; Encrypt + Decrypt
  external.go      # ExternalBackend: subprocess with {{.In}} / {{.Out}} substitution
  mismatch.go      # MismatchChecker: compare stored backend metadata vs. current config
```

`Encryptor.Encrypt(plaintext []byte, outputPath string) ([]byte, error)` â€” encrypts payload; uses `outputPath` for `.sops.yaml` creation-rule matching when no inline recipients are configured.

`Decryptor.Decrypt(ciphertext []byte, inputPath string) ([]byte, error)` â€” decrypts using the operator's available credentials.

`MismatchChecker.Check(encryptedPath string) (mismatch bool, detail string, err error)` â€” reads backend-specific metadata from an existing encrypted file and compares it to the current config.

### `internal/config/`

- Remove the parse-then-error stub for `encryption "sops"` and `encryption "external"`. Decode into `SopsConfig` / `ExternalConfig` structs and surface them in the resolved config.
- Add `LinkCrt []string` to the `CA` struct (HCL attribute `link_crt`). Validation: non-empty list entries, no duplicates, non-empty strings. Relative paths resolve against the config file's directory.

### `internal/plan/`

- New action `WriteEncryptedFile` â€” emitted for key artifacts when backend is not `none`; carries plaintext bytes; `apply` encrypts before writing.
- CA key decryption injected into the signing pre-step: when the resolved CA key path carries the active `output_suffix`, decrypt in-memory via `Decryptor`; no plaintext temp file on disk.
- Mismatch check on every plan build: for each existing encrypted key file, call `MismatchChecker.Check`; accumulate warnings to be printed post-plan.
- New action `CreateSymlink { LinkPath, Target string }` â€” emitted when a declared link is missing, broken, or points to the wrong target.
- New action `DeleteSymlink { LinkPath string }` â€” emitted when a manifest-recorded link is no longer declared in the config.

### `internal/apply/`

- `WriteEncryptedFile`: calls `Encryptor.Encrypt`, writes ciphertext atomically (temp + rename).
- `CreateSymlink`: computes the relative target via `filepath.Rel`, calls `os.Symlink`; creates parent dir if needed.
- `DeleteSymlink`: confirms the path is a symlink via `os.Lstat`; calls `os.Remove` only if confirmed; prints a notice and skips if it is a regular file.

### `internal/manifest/`

- CA record gains `"links": [{"path": "...", "target": "..."}]`.
- Artifact key paths stored with encryption suffix.
- Backend metadata (for mismatch detection) stored per-artifact: `"encryption": {"backend": "sops", "recipients_sha": "..."}` or `"encryption": {"backend": "external", "command_sha": "..."}`. The hash is a stable fingerprint of the active recipients/command, not the key material itself.

### `cmd/nebula-pki/`

- Add `rekey` subcommand: iterates manifest-tracked encrypted artifacts, decrypts each with the stored backend metadata, re-encrypts with the current config, writes atomically, clears mismatch flags. Supports `--dry-run`.
